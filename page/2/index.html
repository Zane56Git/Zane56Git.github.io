拉链表

拉链表最直观区别就是在源表保留历史记录及更新时间；然后这两个很小的记录点能实现很强的数据追溯、回滚功能；

解决了备份数据回溯问题及资源占用问题；

拉链表就是用来存储变化的数据的，每一份数据都有对应的有效期，我们需要进行的操作就是将变动的数据进行新增，同时将变动对应的前一条数据的有效期进行变更。

实现方案

添加创建时间、修改时间字段，记录数据的修改时间及更新时间，如果没有修改过，更新时间记为9999-12-31

表1：订单表（记录原始的数据）
表2：增量数据表（记录每日变更的数据）
表3：历史拉链表（我们要得到的就是这张表）
表4：临时历史拉链表（临时表，处理好之后将数据覆盖到历史拉链表中）

感谢&参考：https://blog.csdn.net/qq_41902618/article/details/109811283

使用说明

数据的使用主要是当前最新的数据和追溯某一天数据：

查询最新的数据

select * from order_his where dw_end_date = ‘9999-12-31’

查询某一天的数据（数据一定会存在有效时间’9999-12-31’，不是最初就最后修改的一条）

select * from order_his where dw_begin_date <= ‘2012-06-21’ and end_date >=’2012-06-21’

基于redis中bigmap的实际应用

发表于 2021-10-29 更新于 2023-06-15 分类于 中间件 Changyan：

基于redis中bigmap的实际应用

redis在实际开发中有很多作用场景，但是实际中只是用了它的基础数据结构。

以下记录一下redis中bigmap高级数据结构及其作用。

bigmap主要是为了减少内存的使用，bigmap就是我们常说的位图结构，主要是

通过设置key不同的offset的值用于记录，值只有0/1两种。

场景:

假设我们需要存储1亿个用户的登录状态，用redis(key，value)对应用户id和状态，假设value为1代表用户已登录，通过查看一个简单的key占用内存为48byte，那么1亿个key所占用内存为:

(48*100000000)/1024/1024/1024=4.47G

这只是一个简单的需求，一天的登录数据量量就占用了这么大内存，明显不可行。

bigmap解决方案:

而如果是bigmap，每一个位是1bit，一亿用户就是一亿bit。8bit=1Byte
100000000/8/1024/1024=11.92M

bitmap基础操作:

1 设置某一位上的值
SETBIT key offset value
offset是偏移量，从0开始，从左往右
2 获取某一位上的值
GETBIT key offset
3 返回指定值0或者1在指定区间上第一次出现的位置
BITPOS key bit [start] [end]

实际运用:

记录一下实际中的一个需求，客户需要后端记录用户在移动端的活动弹窗频率(例如每天一次，每周一次)，其实用户量不多的话完全可以通过redis set记录用户每天参与的活动弹窗记录，然后再进行过滤已展示过的弹窗信息。但如果记录1天一次的key，10w个用户访问就10w个key，100w个用户访问就100w个key，由上面计算分析，其实用户量多就会很浪费资源。

如果我们通过bigmap实现，那我们可以通过不同活动id做为key，以用户id做为offset(前提是用户id是int类型)，因为活动比较少，用户量比较大，这样设计比较合理，假设有100个正在进行的活动，那么每一个活动相当一个bigmap，记录着成千上万的用户的弹窗情况，假设1为已弹窗，0为未弹窗(不存在则为未弹窗)，那么通过bigmap我们大约只需要维护100个key，整体内存也就几十兆，大大减少了内存的占用率。

当然bigmap很有很多常见的运用场景，例如用户签到，用户累计登录情况统计，用户点赞，大数据快速排序等等。后面有时间加强高阶方案的运用了。

CREATE TABLE userbehavior_dwd(

timestamps DATETIME COMMENT ‘行为发生时间戳’ ,
behavior_date VARCHAR(255) COMMENT ‘日期’ ,
behavior_time VARCHAR(255) COMMENT ‘时间’ ,
weekdays VARCHAR(255) COMMENT ‘周-天数’
) COMMENT = ‘用户行为表(维度表)’;

select * from userbehavior_dwd

select min(behavior_date),max(behavior_date) from userbehavior_dwd

delete from userbehavior_dwd where behavior_date>’2017-11-25’

– ▶▶用户行为指标
– 一、按实际维度
– 1、用户活跃度（这里就涉及每天、每时，不会是全量数据，所以会用到天字段或小时字段分组统计）

– 首先将用户活跃度划分为每日活跃度与每时活跃度，并对比UV与PV的情况。

– 日活跃情况
– 我的sql(初步sql，没有参考价值)
select user_id,count(behavior_type) from userbehavior_dwd where behavior_type=’pv’ GROUP BY user_id
– 日活跃情况-实际sql
select behavior_date,count(DISTINCT user_id) as ‘UV’,count(behavior_type) as ‘PV’ from userbehavior_dwd
where behavior_type=’pv’ GROUP BY behavior_date

– 小时活跃情况
select HOUR(behavior_time) as ‘hour’,count(DISTINCT user_id) as ‘UV’,count(behavior_type) as ‘PV’ from userbehavior_dwd
where behavior_type=’pv’ GROUP BY hour

– 人均PV（日）
select behavior_date,count(DISTINCT user_id) as ‘UV’,count(behavior_type) as ‘PV’,
count(behavior_date)/count(DISTINCT user_id) as ‘MeanPVofUV’
from userbehavior_dwd
where behavior_type=’pv’ GROUP BY behavior_date
– 人均PV（小时）同上
– 分析价值：晚上20-22点的PV大幅上涨，是由于人均PV的上升产生的，说明晚上休息时间用户的活跃度相较白天有较大提升。

– 二、用户复购率
– （复购率对于平台来讲是非常重要的行为指标，复购率越高，平台的拉新成本，运营成本越低。）
– 逻辑分析：统计用户购买【相同商品/非相同商品】次数，用户复购率计算逻辑：复购人数/总购买人数
– 2.1、统计复购人数
– 2.2 统计总购买人数
– 2.3 计算复购率

select user_id,count(behavior_type) as ‘buy_num’ from userbehavior_dwd where behavior_type=’buy’ GROUP BY user_id
– 分析复购人员
select t.user_id,if(t.buy_num>1,1,0) as ‘复购人员’
from (select user_id,count(behavior_type) as ‘buy_num’ from userbehavior_dwd where behavior_type=’buy’ GROUP BY user_id ) t

– 统计复购人员数=复购人员总和
select sum(if(t.buy_num>1,1,0)) as ‘复购人数’
from (select user_id,count(behavior_type) as ‘buy_num’ from userbehavior_dwd where behavior_type=’buy’ GROUP BY user_id ) t

– 统计总购买人数
select count(DISTINCT user_id) as ‘总购买人数’ from userbehavior_dwd where behavior_type=’buy’

– 计算复购率=复购人员总和/总购买人数
select count(DISTINCT t.user_id) as ‘总购买人数’,sum(if(t.buy_num>1,1,0)) as ‘复购人数’,
CONCAT(ROUND(0/20*100,2),’%’) as ‘复购率’
from (select user_id,count(behavior_type) as ‘buy_num’ from userbehavior_dwd where behavior_type=’buy’ GROUP BY user_id ) t

– ▶▶购物环节转化率（AARRR分析模型）,主要分为浏览数据转化率、用户数据转化率
– 加购率=浏览数/加购数；购买率=购买数/加购数
– 浏览数据转化率
select pv_sum,cf_sum,buy_sum
,CONCAT(ROUND(cf_sum/pv_sum100,2),’%’) as cp
,CONCAT(ROUND(buy_sum/cf_sum100,2),’%’)as bc
from(
select

– 用户价值指标（RFM分析模型），区分出：价值客户、发展客户、保持客户、挽留客户，一般挽留客户占比最高，进而是价值客户，发展客户，保持客户。
– 1、Recency-上次消费时间（间隔天数），求平均值
– 2、Frequency-消费频率，求平均值
– 3、R、F按价值打分规则&用户分组规则

– r_table、f_table –（TDM）
– r_table
CREATE VIEW R_table(user_id,recentbuy)
as
select u.user_id,
case
when recentbuy is not null then recentbuy
else 10
end as recentbuy
from userbehavior_dwd u left join
(
select user_id
,count(behavior_type) as recentbuy
from userbehavior_dwd
where behavior_type=’buy’
GROUP BY user_id
) t
on u.user_id=t.user_id
– GROUP BY u.user_id
ORDER BY u.user_id

– f_table
CREATE VIEW F_table(user_id,frequency)
as
select u.user_id,
case
when frequency is not null then frequency
else 10
end as frequency
from userbehavior_dwd u left join
(
select user_id
,MIN(DATEDIFF(‘2017-12-08’,behavior_date)) as frequency
from userbehavior_dwd
where behavior_type=’buy’
GROUP BY user_id
) t
on u.user_id=t.user_id
– GROUP BY u.user_id
ORDER BY u.user_id

when r_score<2.35 and f_score<2.04 then ‘挽留客户’
END as userclass
FROM(

(

CASE

when recentbuy<5 and recentbuy>=2 then 3
when recentbuy=1 then 4
END as r_score
FROM r_table
) t
LEFT JOIN
(
SELECT userID,
CASE
when frequency=0 then 1
when frequency<3 and frequency>=1 then 2
when frequency<7 and frequency>=3 then 3
when frequency>=7 then 4
END as f_score
FROM f_table
) f
ON t.userID=f.userID
