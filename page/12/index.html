数据库中索引的运用

发表于 2018-10-24 更新于 2023-06-15 分类于 数据库 Changyan：

MySQL索引的建立对于MySQL的高效运行是很重要的，索引可以大大提高MySQL的检索速度。

数据量达到多少才需要索引

当我们给数据库添加索引的时候，应先了解当前的数据量有多少，到底是数据库量处理的问题还是程序逻辑有影响？

据经验来说，一般超过10万条就应该考虑索引，当然如果几万条数据查询慢的话也可以加上索引（例如，数据量不大，但是查询通过in很多参数，如果有索引就轻松很多）。

索引操作记录

以下记录一下mogondb的索引优化记录

今天处理同事开发的一个模块，查看了kafka消费速率很慢，然后发现是数据处理时操作mongo有影响，由于之前说有网络延迟影响，调试也麻烦就没细查，联系网络相关人员排查无果，后面通过代码调试才发现是有一个查询方法过慢，一个查询余约24秒，最后通过添加索引进行优化，消费速率大大提升。

mongo 125608条（数据量不大，不同数据库数量级如上）

  • 没索引，组合查询（用到两个条件）

耗时24492ms（24.492s）

  • 优化后，两个条件字段都加上索引

提升了约4千倍，这就是差距！！！

了解自己的系统，不要过早优化!

比如：几千条数据的新闻表，每天几百几千次的正文搜索，大多数时候我们可以放心的去like，而不要又去建一套全文搜索什么的，毕竟cpu还是比人脑厉害太多

Java-设计模式之七大原则

发表于 2018-09-16 更新于 2024-05-24 分类于 JAVA Changyan：

设计模式是一种设计思想，可能有些设计模式的运用场景相似，只能说业务场景可以被多种设计模式实现，不能说这种业务场景就是需要某一种设计模式实现。

如果没用设计模式，代码很多就是通过if else实现，业务复杂了就回显得很臃肿；

七大原则

  1. 单一原则

（业务功能逻辑不能混）
在类级别增删，在方法级别增删

（需在类级别遵守单一职责，类方法数量少的情况下，可在方法级别遵守单一职责）

  1. 接口隔离原则（拆分原则）

一个类对另一个的依赖建立在最小的接口上（为了降低接口业务耦合）

按接口职责进行隔离拆分，可以理解为拆分原则

（接口方法按业务及实现类拆分接口）

例如一个接口有5个方法，但业务类A只需要使用接口里面的其中2个方法，并且使用别的接口中的接口；显然业务类A并不需要接口的所有并且包含无用的接口，解决方案就是需要遵守“接口隔离原则”：

将接口拆分为多个接口，业务类可以实现多个接口（只实现所需的接口）

  1. 依赖倒置原则

核心思想：面向接口编程

1、接口当参数传递
（以接口为参数，现在都是通过依赖注入接口后，直接调用）

类似策略模式，通过工厂类（方式很多。例如StringUtils获取bean）获取策略器（接口），不同策略器实现不同的接口方法逻辑，所以在调用方法其实是面向接口的，不同的策略实现类执行其对应的接口方法。

2、构造方法传递

3、set方法传递

  1. 里氏替换原则

继承带来了便利，也带来了代码入侵的问题

主要针对继承设定的代码规范

父类的变化会影响各个子类，子类尽量不要重写父类的方法

(无意改造了父类方法导致计算异常，本例子有点牵强，但也能说明其影响点)

主要将继承换成类的组合

  1. 开闭原则

扩展开放，修改关闭
（对提供方开放扩展，对使用方关闭修改）；
类似设计模式不同的业务类型只需要进通过继承和实现扩展新的功能，不修改原先已存在的业务类；

用抽象构建框架，用实现扩展细节

  1. 迪米特法则

核心：降低类之间的耦合。
只与直接的朋友通讯（最直接的关系），将陌生类方法封装成方法进行调用，而不是将其业务逻辑写在主业务代码里面。

  1. 合成复用原则

核心思想：尽量用合成和聚合的方式，而不是继承

Spring Data JPA

jpa文档

JPA(Java Persistence API)意即Java持久化API，是Sun官方在JDK5.0后提出的Java持久化规范(一套基于ORM的规范),（JSR 338，这些接口所在包为javax.persistence，详细内容可参考https://github.com/javaee/jpa-spec）
ORM:对象关系映射（英语：Object Relational Mapping，简称ORM，或O/RM，或O/R mapping），是一种程序设计技术，用于实现面向对象编程语言里不同类型系统的数据之间的转换。

dao层
一个表对应一个dao层接口，该接口要继承 org.springframework.data.jpa.repository.JpaRepository<T, ID> 接口：
第一个泛型：实体类
第二个泛型：实体类中Id（数据库主键）的类型

核心概念

Spring Data 存储库抽象中的中心接口是Repository. 它需要域类来管理以及域类的 ID 类型作为类型参数。此接口主要用作标记接口，以捕获要使用的类型并帮助您发现扩展此接口的接口。该CrudRepository接口为正在管理的实体类提供复杂的 CRUD 功能。

Spring Data JPA的查询方式及方法名查询规则

使用JPQL的方式查询

  1. JPQL查询
    @Query 注解的使用非常简单，只需在方法上面标注该注解，同时提供一个JPQL查询语句即可

  2. 原始sql（nativeQuery=true）
    @Query(value=”select * from table_name”,nativeQuery=true)

  3. 修改
    用 @Modifying 来将该操作标识为修改查询，这样框架最终会生成一个更新的操作，而非查询
    做修改的时候:用自己实现的修改和删除,必须要加上事务,且JPA会自动回滚,需要收到加上@Rollback(value= false),设置为不自动回滚

基于内容的推荐

基于内容推荐的原理是根据用户感兴趣的物品A，找到和A内容信息相近的物品B，从而将B推荐给A。

而提取用户偏好的物品特征是基于内容推荐算法的关键，基于内容的推荐过程是用户喜欢的物品和特征的描述。

物品的特征有属性、描述等，图书的特征是一些文本内容，特征提取可能涉及文本处理相关技术，将文本内容转化为可计算的向量形式，实现对物品的特征建模，应用推荐算法进行内容推荐。

除此之外，还有相似度计算（根据关键词进行相似度计算，常用的算法有TF-IDF）。基于内容的推荐优点是简单有效，推荐结果直观，容易理解，不需要领域知识。不需要用户的历史行为数据，比如对物品的评价等。

https://blog.csdn.net/legendaryhaha/article/details/90573238

协同过滤算法

协同过滤 CF（Collaborative Filtering）简单来说是利用某兴趣相投、拥有共同经验之群体的喜好来推荐用户感兴趣的信息，个人通过合作的机制给予信息相当程度的回应（如评分）并记录下来以达到过滤的目的进而帮助别人筛选信息，回应不一定局限于特别感兴趣的，特别不感兴趣信息的纪录也相当重要。

协同过滤推荐算法分为两类，分别是基于用户的协同过滤算法(user-based collaboratIve filtering)，和基于物品的协同过滤算法(item-based collaborative filtering)。简单的说就是：人以类聚，物以群分。
